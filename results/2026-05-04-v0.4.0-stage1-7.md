## Provenance block (Stage 1, all docs)

Every per-doc JSON now carries:

```json
"provenance": {
  "original_form": "digital_native | image_only_pdf | paper_scan | photo",
  "source_engine": "pdfplumber-0.11.9 | pypdfium2-5.7.1+ocr | cv2.imread+ocr",
  "file_sha256": "<64 hex>",
  "file_size_bytes": <int>,
  "ingest_timestamp_utc": "<RFC3339>",
  "page_count": <int>,
  "pdf_metadata": { "Title": "...", "Author": "...", "CreationDate": "...", ... },
  "vatca_attestation": "original-form-preserved"
}
```

This satisfies the v0.4 PRD D-V0.4-04 IE compliance moat ‚Äî no competitor surfaces VATCA Section 84(3) original-form attestation at extraction time.

## Open questions surfaced by v0.4.0 bench

1. **LLM extract on Vega 8 = 78s cold, ~30s warm.** PRD ¬ß8 Open Question 1 confirmed: at 5 customers √ó 200 docs/mo = 1,000 docs/mo, full LLM-extract path = ~25 hours/mo of pure inference time on this CPU. Acceptable for v0.4.0 ship; flag for v0.4.x where LoRA-on-3B-distilled may halve.
2. **LLM extract redundant on native PDFs.** Voter is already at 1.0 confidence after pdfplumber; LLM call adds latency without lifting accuracy. Suggest: skip LLM extract when `original_form == "digital_native"`. Implement in v0.4.1.
3. **HaluGate token-logprob proxy is field-confidence only.** Real token logprobs require switching from instructor to native Ollama `/api/generate` with stream-token capture. Deferred v0.4.x.
4. **PaddleOCR PP-StructureV3 (Stage 4 table layer)** not yet wired ‚Äî invoice line-item extraction blocked until shipped. v0.4.1 priority.
