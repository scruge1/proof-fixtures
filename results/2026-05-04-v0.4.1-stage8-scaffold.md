## What this is

Stage 8 of the v0.4 PRD is the corrections flywheel: when a doc bounces below
the 0.98 confidence gate (or when a customer disputes an accepted extraction),
a reviewer corrects the fields in Label Studio CE; each correction lands in
the portal Postgres `corrections` table, fires a `NOTIFY corrections_channel`,
and the consumer daemon appends the correction to a DVC-tracked JSONL corpus
that feeds the v0.5 LoRA / distillation training run.

Per CALLMEIE-DOCAI-V0.4-PRD.md D-V0.4-06 (Label Studio CE) + D-V0.4-07
(Hetzner Object Storage + DVC).

## Schema (extractions + corrections)

```
extractions (UUID PK, tenant_id FK, doc_id, file_sha256, source_path, schema_version,
             pipeline, original_form, source_engine, page_count, vatca_attestation,
             ingest_timestamp_utc, payload JSONB, gate_passed, bounce_reason, created_at)
  + uniq(tenant_id, file_sha256)        — idempotent ingest
  + idx(tenant_id, doc_id) / sha256 / (gate_passed, created_at)

corrections (UUID PK, tenant_id FK, extraction_id FK, user_id FK, field_id,
             original_value, corrected_value, reason, evidence_bbox FLOAT[],
             label_studio_task_id, submitted_at)
  + idx(extraction_id) / (tenant_id, submitted_at) / field_id

notify_correction()  TRIGGER → pg_notify('corrections_channel', NEW.id::text)
```

`field_id` is `vendor` / `total` / `vat` / `date` for top-level field corrections,
or `line_items[<row_idx>].<sub_field>` for line-item corrections (`description`,
`quantity`, `unit_price`, `amount`).

`evidence_bbox` is the corrector's bounding-box on the source image (Label Studio
emits this when the reviewer draws the correction region).

## Consumer behaviour

```
LISTEN corrections_channel
on NOTIFY <correction_id>:
    SELECT correction + extraction context  →  JSONL append
fallback poll every POLL_TIMEOUT_SEC (default 300s) — catches missed NOTIFYs
```

Backfill mode (`--backfill`) replays all corrections to a fresh JSONL — for
when DVC corpus needs rebuild after a schema change.
