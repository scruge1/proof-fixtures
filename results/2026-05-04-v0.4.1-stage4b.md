## Why rapid-table over PaddleOCR PP-StructureV3 directly

PaddleOCR 3.5 + paddlepaddle 3.3 fails on Windows runtime with:

```
NotImplementedError: ConvertPirAttribute2RuntimeAttribute not support
[pir::ArrayAttribute<pir::DoubleAttribute>] (at onednn_instruction.cc:118)
```

This is the documented "Windows install pain" risk-flag from
`research/2026-05-04-hybrid-stack-libraries.md` §4.7. The PIR (new IR) path
in PaddlePaddle 3.x with onednn is incomplete on Windows.

`rapid-table` exports the same PP-Structure ONNX weights without the
paddlepaddle runtime — works on Vega 8 / Windows / no CUDA in 2.5-3s/page.
For Linux / Hetzner deployment, PaddleOCR PP-StructureV3 directly may be
preferred (better layout + reading-order recovery on dense receipts).

Mitigation per research: "Use Linux container for dev too; AX52 is Linux
anyway." Implemented mitigation: ONNX bridge via rapid-table for Windows dev.

### Line items extracted

Test fixture `test-invoice-001.png` (Chadwicks invoice with Cement + Sand line items):

```json
"line_items": [
  { "description": null, "amount": 145.5, "raw_cells": ["EUR 145.50"], "row_idx": 9 },
  { "description": "Sand (1m3)", "quantity": 1.0, "amount": 85.0,
    "raw_cells": ["Sand (1m3)", "EUR 85.00"], "row_idx": 10 }
]
```

**Known gap:** PP-Structure split the Cement row into two cells — description
"Cement (10x bags)" on one row and "EUR 145.50" on the next. The current row
parser doesn't merge adjacent rows when one has only money. This drops the
description for row 9. Sand row was correctly merged.

Refinement plan (v0.4.2): Y-coordinate clustering on `cell_bboxes` to merge
adjacent rows that look like a single line item split across two grid lines.

## Open questions surfaced

1. **PP-Structure row segmentation on receipts without ruled lines.** Adjacent
   rows split when they shouldn't. Y-coordinate merge heuristic is v0.4.2
   priority. PaddleOCR LayoutDetection pre-pass to crop to table region only
   is a deeper fix.
2. **Header detection vs implicit-header invoices.** Many IE retailer receipts
   have no "Description / Qty / Amount" header — the line-item table is
   implicit. Current `_LINE_ITEM_HEADER_HINTS` requires ≥2 keyword matches.
   When that fails, all rows are candidates. Mitigation: drop rows with no
   money-shaped cell (already implemented); but description-only sub-rows
   still get dropped.
3. **PaddleOCR Linux native install for Hetzner deployment.** Plan:
   container-build paddleocr+paddlepaddle on the AX52 image; rapid-table
   stays as Windows-dev fallback only.
